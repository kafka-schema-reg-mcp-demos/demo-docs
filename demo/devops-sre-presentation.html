MCP is a standardized protocol that enables AI applications to securely connect with various data sources and services through a unified interface.

MCP Technical Architecture

🔌 Protocol Layer

  • JSON-RPC 2.0 based communication
  • Supports multiple transport mechanisms
  • Bi-directional streaming capabilities

🔒 Security Layer

  • OAuth 2.0 authentication
  • Fine-grained permission scopes
  • Encrypted transport (TLS)

📊 Resource Layer

  • Unified resource access
  • Schema validation
  • Version management

🎯 Tool Layer

  • Standardized tool definitions
  • Parameter validation
  • Error handling

Key Insight for DevOps

MCP provides a standardized, secure bridge between AI models and your infrastructure - think of it as "OAuth for AI" with built-in observability.

MCP Server Deployment Modes

Local vs Remote Hosting

🖥️ Local Mode

Characteristics:

  • Runs on user's personal computer
  • Direct process spawning via stdio
  • No network exposure
  • Simple transport layer

Security & Responsibility:

  • ✅ User controls entire stack
  • ✅ Data never leaves local machine
  • ⚠️ User responsible for security
  • ⚠️ Manual updates & maintenance

Best For:

  • Personal productivity tools
  • Development & testing
  • Sensitive data processing

☁️ Remote Mode

Characteristics:

  • Hosted in private network or internet
  • HTTP/WebSocket transport
  • Multi-tenant architecture
  • Shared infrastructure

Security Requirements:

  • 🔒 Authentication (OAuth 2.0/JWT)
  • 🔒 TLS/SSL encryption
  • 🔒 API rate limiting
  • 🔒 Data isolation between users
  • 🔒 Audit logging & monitoring

Best For:

  • Team collaboration
  • Enterprise deployments
  • Scalable services

⚠️ Important Consideration

Remote MCP servers require significant additional infrastructure for security, multi-tenancy, and reliability - transforming from a simple stdio protocol to a full distributed system with authentication, authorization, and resource management.

Current State: MCP 2025-06-18

Latest Specification Updates

✅ What's New

  • JSON-RPC compliance: Full adherence to JSON-RPC 2.0 specification
  • No batching: Simplified protocol, one request per message

⚠️ Breaking Changes

  • Removed JSON-RPC batching support
  • Updated header requirements
  • New scope definitions for OAuth

Security Concerns & Mitigation

Addressing LLM Hallucinations in Production

🚨 The Hallucination Risk

LLMs can generate plausible but incorrect commands that could harm production systems:

  • Incorrect database queries leading to data loss
  • Misconfigured infrastructure changes
  • False positive security responses

✅ MCP's Security Solutions

  • Schema Validation: All requests validated against predefined schemas
  • Permission Scoping: Fine-grained OAuth scopes limit access
  • Audit Logging: Complete request/response tracking
  • Dry-run Mode: Test changes before execution
  • Human-in-the-loop: Critical operations require approval
// Example: Safe operation with validation
{
  "method": "database/query",
  "params": {
    "query": "SELECT * FROM users LIMIT 10",
    "dry_run": true,
    "require_approval": true,
    "max_rows": 10
  }
}

Self-Hosted LLMs & MCP

Working with Smaller Models Safely

Why Self-Hosted Models?

  • Data sovereignty and compliance requirements
  • Cost optimization for high-volume operations
  • Reduced latency for on-premise deployments
  • Custom fine-tuning for domain-specific tasks

📊 Model Considerations

  • 7B-13B parameters optimal for most ops
  • Llama, Mistral, or specialized models
  • Fine-tuning on operational data
  • Quantization for efficiency

🛡️ Safety Mechanisms

  • Constrained action spaces
  • Template-based responses
  • Validation layers
  • Rollback capabilities

SLIM_MODE: Optimizing for Self-Hosted Models

⚡ Performance Challenge

Smaller models (7B-13B) have limited context windows and processing power

✅ SLIM_MODE Solution

  • Reduced response sizes: Concise, focused outputs
  • Simplified schemas: Essential fields only
  • Batched operations: Minimize round trips
  • Caching layer: Reduce redundant processing
// SLIM_MODE configuration
{
  "slim_mode": {
    "enabled": true,
    "max_response_tokens": 512,
    "schema_simplification": true,
    "response_format": "concise",
    "batch_size": 10,
    "cache_ttl": 3600
  }
}

Results

  • 70% reduction in response size
  • 3x faster processing with 7B models
  • Maintained accuracy with validation
