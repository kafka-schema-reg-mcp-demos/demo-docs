MCP Specification 2025-06-18

Latest Protocol Evolution & Breaking Changes

🚀 MCP is less than one year old - Introduced by Anthropic in November 2024, it's rapidly evolving with significant updates every few months.

🔄 Protocol Changes

  • ❌ Removed JSON-RPC batching - Breaking from JSON-RPC 2.0 standard
  • Structured tool output - Tools return predictable JSON
  • MCP-Protocol-Version header - Required for all HTTP requests
  • Resource links - Reference URIs instead of inlining

🔐 Security Enhancements

  • OAuth Resource Servers - MCP servers classified as OAuth 2.1 resources
  • RFC 8707 compliance - Resource indicators prevent token theft
  • PKCE required - Protection against authorization code interception
  • Security best practices - New dedicated documentation

💬 New Capabilities

  • Elicitation - Servers can ask users for additional info mid-session
  • Roots - Server-initiated filesystem boundary inquiries
  • Sampling - Server-initiated recursive LLM interactions
  • Progress tracking - Better async operation monitoring

⚠️ Migration Considerations

  • Breaking compatibility - 2025-03-26 clients may not work with 2025-06-18 servers
  • SDK updates required - JSON-RPC libraries need modifications
  • Header validation - Servers must check protocol version
  • Backward compatibility - Default to 2025-03-26 if header missing

⚠️ Critical for DevOps

The removal of JSON-RPC batching is controversial - it breaks standard JSON-RPC 2.0 compliance. If you're using standard JSON-RPC libraries, you'll need to disable batching or modify your implementation. Consider maintaining version compatibility matrices for your MCP deployments.

  • Bi-directional streaming capabilities

🔒 Security Layer

  • OAuth 2.0 authentication
  • Fine-grained permission scopes
  • Encrypted transport (TLS)

📊 Resource Layer

  • Unified resource access
  • Schema validation
  • Version management

🎯 Tool Layer

  • Standardized tool definitions
  • Parameter validation
  • Error handling

Key Insight for DevOps

MCP provides a standardized, secure bridge between AI models and your infrastructure - think of it as "OAuth for AI" with built-in observability.

MCP Server Deployment: Local vs Remote

🖥️ Local Mode

Characteristics

  • Direct process spawning via stdio
  • Runs on user's machine
  • Simple transport layer
  • Native OS integration

Security & Risks

  • ✅ User controls entire stack
  • ✅ No network exposure
  • ⚠️ User responsible for security
  • ⚠️ Resource consumption on local machine
  • ⚠️ Manual updates & maintenance

☁️ Remote Mode

Characteristics

  • Network-based communication
  • Shared infrastructure
  • Requires transport protocol (HTTP/WebSocket)
  • Multi-tenant architecture

Security & Risks

  • 🔒 Authentication & authorization required
  • 🔒 Data isolation between tenants
  • ⚠️ Network security concerns
  • ⚠️ Rate limiting & resource quotas
  • ⚠️ Compliance & data residency

🔧 Making MCP Server Remote-Ready

Transport Layer

  • HTTP/HTTPS endpoints
  • WebSocket for bidirectional
  • gRPC for performance
  • TLS/SSL encryption

Security Layer

  • OAuth 2.0 / JWT tokens
  • API key management
  • Role-based access control
  • Audit logging

Multi-tenancy Layer

  • Tenant isolation
  • Resource quotas
  • Usage metering
  • Data partitioning

Operations Layer

  • Load balancing
  • Health checks
  • Monitoring & alerting
  • Scaling policies

Key Insight: Remote MCP servers require significant additional infrastructure for security, multi-tenancy, and reliability - transforming from a simple stdio protocol to a full distributed system.

Security Concerns & Mitigation

Addressing LLM Hallucinations in Production

🚨 The Hallucination Risk

LLMs can generate plausible but incorrect commands that could harm production systems:

  • Incorrect database queries leading to data loss
  • Misconfigured infrastructure changes
  • False positive security responses

✅ MCP's Security Solutions

  • Schema Validation: All requests validated against predefined schemas
  • Permission Scoping: Fine-grained OAuth scopes limit access
  • Audit Logging: Complete request/response tracking
  • Dry-run Mode: Test changes before execution
  • Human-in-the-loop: Critical operations require approval
// Example: Safe operation with validation
{
  "method": "database/query",
  "params": {
    "query": "SELECT * FROM users LIMIT 10",
    "dry_run": true,
    "require_approval": true,
    "max_rows": 10
  }
}

Self-Hosted LLMs & MCP

Working with Smaller Models Safely

Why Self-Hosted Models?

  • Data sovereignty and compliance requirements
  • Cost optimization for high-volume operations
  • Reduced latency for on-premise deployments
  • Custom fine-tuning for domain-specific tasks

📊 Model Considerations

  • 7B-13B parameters optimal for most ops
  • Llama, Mistral, or specialized models
  • Fine-tuning on operational data
  • Quantization for efficiency

🛡️ Safety Mechanisms

  • Constrained action spaces
  • Template-based responses
  • Validation layers
  • Rollback capabilities
# Example: SLIM_MODE configuration
export MCP_SLIM_MODE=true
export MCP_MAX_RESPONSE_SIZE=1024
export MCP_VALIDATION_STRICT=true
export MCP_REQUIRE_SCHEMAS=true
